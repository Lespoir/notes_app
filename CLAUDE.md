# CLAUDE.md — Notes App

## Project Overview

Monorepo for a notes application.

- **Frontend:** Next.js (App Router) at `apps/web/`
- **Backend:** Django API at `apps/api`
- **API contract:** Backend generates OpenAPI spec via `drf-spectacular`; frontend generates TanStack Query hooks via Orval
- **Design system:** Custom `notesDS` with Tailwind CSS + CVA variants

---

## Frontend (`apps/web/`)

**Path alias:** `@/*` resolves to `apps/web/src/*`

### Directory Structure (`src/`)

```
src/
  app/            # Routing layer (Next.js App Router) — NO logic
  data/           # Auto-generated API hooks + DTOs — NEVER edit manually
  domains/        # Business logic (entities, repositories, rules, schemas)
  features/       # UI layer (screen hooks + components, one per screen)
  notesDS/        # Design system (pure UI, no business knowledge)
  lib/            # Cross-cutting concerns (auth, query client, monitoring)
```

### Import Rules

Strict unidirectional dependency flow. Violations break the architecture.

| Layer          | Can import from                     | NEVER imports from                  |
|----------------|-------------------------------------|-------------------------------------|
| `app/`         | `features/`                         | `domains/`, `data/`, `notesDS/`     |
| `features/`    | `domains/`, `notesDS/`              | `data/`                             |
| `domains/`     | `data/`, `lib/`                     | `features/`, `notesDS/`             |
| `data/`        | (external libs only)                | anything above it                   |
| `notesDS/`     | (only itself + tailwind)            | `domains/`, `features/`, `data/`    |
| `lib/`         | (external libs only)                | `domains/`, `features/`             |

```
app/ -> features/ -> domains/ -> data/
                  -> notesDS/
```

### Data Flow

```
app/page.tsx -> features/*/hooks/useScreen.ts -> domains/*/repositories/*.repository.ts -> data/generated/[tag]/[tag].ts
```

- **Pages** call a screen hook and pass data to components. No logic in pages.
- **Screen hooks** (`use[ScreenName].ts`) orchestrate domain repositories and manage presentation state.
- **Repositories** wrap Orval-generated hooks, transform DTOs to entities, handle cache invalidation.
- **Rules** (`domains/*/rules/`) contain pure business logic functions.
- **Schemas** (`domains/*/schemas/`) contain Zod validation for inputs.

### Do's

- Use Zod schemas in `domains/*/schemas/` to validate all inputs
- Use repositories for ALL data access — never call generated hooks from features
- Use domain rules (`domains/*/rules/`) for business calculations — keep them pure
- Use layout primitives (`Stack`, `Row`, `Container`, `Section`, `CardShell`) — no raw `<div>` soup
- Use typography components (`H1`-`H4`, `P`, `Large`, `Small`, `Muted`, `Lead`, `Overline`) — no raw heading/paragraph tags
- Use CVA (`class-variance-authority`) for all multi-variant components
- Export both the component and its `*Variants` const from design system components

### Don'ts

- Don't write business logic in `.tsx` files or screen hooks — extract to domain rules
- Don't import DTOs from `@/data/` in features — use entities from `@/domains/*/entities/`
- Don't use `useEffect` for data fetching — use React Query via repositories
- Don't import from `@/features/` in domains — domains must not know about UI
- Don't edit files in `data/generated/` — they are auto-generated by Orval
- Don't use raw `<div>` for layout structure — use notesDS primitives
- Don't use ternary-chained Tailwind strings — use CVA variants
- Don't hardcode color values — reference `@theme` tokens only

## Backend (`apps/api/`)

Django 6.0 + Django REST Framework. Config module at `apps/api/config/`.

### Module Structure

Each Django app follows this layout:

```
<module>/
  models.py
  actions/          # Write operations (create, update, delete) + business invariants
  readers/          # Read-only queries for API responses (filters, ordering, pagination)
  lookups/          # Minimal reads for action decisions (existence checks, FK lookups)
  interfaces/
    api/
      urls.py
      views.py
      schemas.py    # DRF serializers (input validation + output serialization)
      permissions.py
  tests/
```

### Import Rules

DRF is transport only — it stays inside `interfaces/api/`.

| Layer                | Can import DRF | Can import from                        |
|----------------------|----------------|----------------------------------------|
| `interfaces/api/*`   | Yes            | `actions/`, `readers/`, `lookups/`     |
| `actions/`           | No             | `lookups/` (own + other modules)       |
| `readers/`           | No             | models, ORM only                       |
| `lookups/`           | No             | models, ORM only                       |

### Data Flow

```
Request -> interfaces/api/views.py -> actions/ or readers/ -> models/ORM
```

- **Views** are thin: validate input (serializer), call action/reader, serialize output
- **Actions** handle writes + business invariants + explicit side effects
- **Readers** handle read queries, own performance (`select_related`/`prefetch_related`)
- **Lookups** are lightweight reads for action decisions (`values_list`/`only`)

### Error Handling

- Business code raises domain exceptions (`BusinessError` subclasses) — no HTTP concepts
- API layer maps exceptions to HTTP status codes via centralized exception handler

| Exception             | HTTP Status |
|-----------------------|-------------|
| `NotFoundError`       | 404         |
| `PermissionDeniedError` | 403       |
| `ConflictError`       | 409         |
| `ValidationError`     | 422         |

### Naming Conventions

- **Actions:** verb-first — `create_*`, `update_*`, `delete_*`, `assign_*`
- **Readers:** noun-first — `get_*`, `list_*`, `search_*`
- **Lookups:** `find_*` — `find_*_id_by_*`

### OpenAPI Schema Generation

The backend uses `drf-spectacular` to auto-generate the OpenAPI spec that the frontend consumes via Orval.

- Every view must be annotated with `@extend_schema` to ensure accurate request/response types
- Serializers in `schemas.py` drive the generated spec — keep them precise
- The generated spec is the contract between backend and frontend; inaccurate schemas break client generation

### Do's

- Annotate all views with `@extend_schema` (request, response, parameters)
- Keep views thin: validate, delegate, serialize
- Use `select_related`/`prefetch_related` in readers to avoid N+1
- Use `values_list`/`only` in lookups for lightweight queries
- Make side effects explicit in actions
- Return QuerySets from readers when possible — let DRF serialize
- Write unit tests for actions/lookups + API tests for endpoint contracts

### Don'ts

- Don't put business logic in views or serializers
- Don't use Django signals for core workflows — side effects must be explicit in actions
- Don't import DRF outside `interfaces/api/`
- Don't use HTTP status codes outside the API layer
- Don't put complex ORM queries in views — they belong in readers
- Don't put business logic in serializer `create()`/`update()` methods
- Don't leave views without `@extend_schema` — the frontend depends on the generated OpenAPI spec
